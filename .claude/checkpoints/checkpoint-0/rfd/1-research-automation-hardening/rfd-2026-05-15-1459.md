# RFD: Research Automation Hardening

**Date**: 2026-05-15
**Status**: Implemented
**Author**: Claude (Opus 4.7) for Pouria Rouzrokh

---

## Question

The nightly Google Scholar citation update has been running for ~30 days. I want to
audit the system now that it has been operational for a while, eliminate any silent
data/disk accumulation, make it resilient to transient failures, and make the run
frequency configurable through a single source of truth — so it stays healthy through
year-long unattended operation.

---

## Audit Findings (2026-05-15)

### Working as intended
- Cron fires daily at 02:00 UTC and a successful headless Claude run completes in ~5–9 min.
- `logs/run_<timestamp>/` directories are properly rotated by `mtime +7`.
- Per-run logs and screenshots are written, plus an append-only `logs/cron.log` summary.
- `logs/`, `.next/`, `.playwright-mcp/`, `.env*` all gitignored.

### Silent problems that would compound over a year

1. **`.playwright-mcp/` accumulates without cleanup.** 128 files, 6.2 MB at audit time
   with files dating back to 2026-02-15 (~3 months). The MCP server writes a console
   log + page snapshot per session; nothing was reaping them. Linear growth in file count.
2. **Dirty working tree blocks `git pull --rebase` every single night.** Three stale
   modifications (deleted RFD, modified `CLAUDE.md`, modified `snapshot.md`) caused
   the daily `git pull` to fail with `cannot pull with rebase: You have unstaged changes`.
   The script swallowed the error and proceeded; this hid the problem but introduced
   real risk: any concurrent remote push during a dirty period would cause the
   headless agent's `git push` to fail and the day's data update to be lost.
3. **`test-results/` was not gitignored.** Empty but polluted `git status`.
4. **All operational knobs were scattered.** Schedule lived in crontab; model
   (`claude-opus-4-6`), cooldown (79200 s), and retention (7 d) were hardcoded
   constants in `update-research-cron.sh`. No central place to tune them, and the
   hardcoded model name is a time bomb when Anthropic sunsets that model.
5. **No retry on transient failure.** A single Google Scholar CAPTCHA or network
   blip lost the entire day's update.
6. **No "current status" surface.** The only way to verify health was to grep
   `cron.log` or open the latest `run_*/update.log`.
7. **No max-runtime on the `claude` invocation.** A hung session could block the
   22-hour cooldown and silently take the system out of service.
8. **`cron.log` was append-only with no rotation.** Slow grower (~3–4 lines/day,
   ~120 KB/year) but unbounded.

### Confirmed non-issues
- `cron.log` size at audit: 52 KB (52 days of history). Steady.
- Steady-state `logs/` size: ~22 MB (9 days × 2.5 MB/run). Steady.
- The single `FAILED (exit 143)` in cron.log on 2026-05-14 20:59:22 UTC was an
  unrelated manual/scheduled run that received SIGTERM; the next regular 02:00
  cron succeeded. Not a real failure of the automation.

---

## Plan

Consolidate operational config, add resilience primitives, and shed silent
accumulation. Keep the headless agent prompt and the existing cron schedule
unchanged.

### Files

| Path | Action | Purpose |
| --- | --- | --- |
| `utils/research-update.config.json` | NEW | Single source of truth: schedule, model, cooldown, retries, retention, max runtime, cron-log rotation threshold |
| `utils/update-research-cron.sh` | Rewrite | Reads config; adds auto-stash before pull, retries with delay, `timeout` wrapper, `.playwright-mcp/` cleanup, longer retention for failed runs, cron-log rotation, `STATUS.md`, `history.jsonl`, `--dry-run` mode |
| `utils/setup-research-cron.sh` | Rewrite | Reads schedule from config and syncs crontab; idempotent |
| `.gitignore` | Update | Add `test-results/` |
| `utils/update-research-prompt.md` | Unchanged | Prompt is working |

### Pending working-tree commit (separate)

Committed first as `Commit pending docs cleanup` (commit `d4cc874`) so the daily
`git pull --rebase` immediately stops failing — independent of the script rewrite.

### Resilience features added

- **Auto-stash before pull.** If the tree has uncommitted or untracked changes,
  the script does `git stash push --include-untracked -m research-update-autostash-<ts>`,
  then `git pull --rebase`, then `git stash pop` for that specific stash. If the
  pop conflicts, the stash is kept and the failure is logged loudly — the user
  can resolve manually without losing data.
- **`timeout` wrapper** around `claude` invocation using `max_runtime_seconds`
  from config (default 1800 s = 30 min). Sends TERM, then KILL 30 s later.
- **Retries.** `max_retries` attempts (default 2 = 3 total tries) with
  `retry_delay_seconds` (default 600 s = 10 min) between attempts.
- **Tiered retention.** Successful runs retained `log_retention_days` (default 7);
  failed runs retained `failure_log_retention_days` (default 30) — failures get
  more time to be debugged. Detected via a `FAILED` marker file in the run dir.
- **`.playwright-mcp/` cleanup** every run by `mtime > playwright_mcp_retention_days`.
- **`cron.log` rotation** when it exceeds `cron_log_max_bytes` (default 1 MB);
  keeps the most recent half.

### Observability surface

- **`logs/STATUS.md`** — rewritten every run. Human-readable "last run" snapshot.
- **`logs/history.jsonl`** — one JSON line per run; ~365 lines/year. Cheap to grep
  for failure streaks, durations, etc.
- **Per-run `update.log`** — unchanged.
- **`logs/cron.log`** — unchanged behavior, now rotated.

### Configurability

`utils/research-update.config.json`:

```json
{
  "schedule": "0 2 * * *",
  "model": "claude-opus-4-7",
  "cooldown_seconds": 79200,
  "max_runtime_seconds": 1800,
  "max_retries": 2,
  "retry_delay_seconds": 600,
  "log_retention_days": 7,
  "failure_log_retention_days": 30,
  "playwright_mcp_retention_days": 7,
  "cron_log_max_bytes": 1048576
}
```

To change the schedule (or any knob): edit the JSON, then re-run
`bash utils/setup-research-cron.sh` to sync the crontab.

### Decisions explicitly considered and made

- **Default model upgraded** from `claude-opus-4-6` to `claude-opus-4-7`.
- **No external alerting.** Per user request, failures surface through the
  on-disk `STATUS.md` + `history.jsonl` + `cron.log`. No email, webhook, or
  external service introduced.
- **Auto-stash is silent on success, loud on conflict.** The user accepted this
  trade-off so dirty local edits never block the automation.
- **`--dry-run` mode added** to validate config and exercise cleanup logic
  without invoking the model.
