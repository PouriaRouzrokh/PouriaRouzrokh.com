# RFD: Nightly Deployment Failure — Diagnosis, Fix, and Deploy Verification

**Date**: 2026-06-18
**Status**: Implemented
**Author**: Claude (Opus 4.8) for Pouria Rouzrokh

---

## Question

For three consecutive nights (Jun 16, 17, 18) the nightly research update committed and
pushed valid data, but the Vercel production deployment failed. The cron reported
`success` every night and nothing surfaced the problem. Find the real cause, fix it,
make sure a deploy actually succeeds, and make sure a failed deploy can never again pass
silently.

---

## Diagnosis

The reported symptom ("invalid JSON") was a red herring. Evidence gathered:

- `public/content/research.json` parsed as valid JSON for every recent commit, and a
  local `npm run build` of the exact failing commit (`b655e17`) succeeded.
- A clean `npm install` succeeds because the committed `.npmrc` already sets
  `legacy-peer-deps=true`, so the ESLint peer-dependency conflict does **not** break
  Vercel's install.
- The failed Vercel deployments errored **immediately** (`ready == buildingAt ==
  createdAt`), produced **no build logs**, and lacked the `bundler` / `lambdaRuntimeStats`
  metadata that successful deployments carry — i.e. the build never ran.
- The GitHub "Vercel" commit status for the failed commits was `failure` with
  `target_url` pointing at **`vercel.com/docs/.../project-configuration`** (the
  successful Jun 15 commit pointed at the deployment inspector). That target_url is the
  link Vercel attaches when it rejects a deployment for a `vercel.json` **configuration**
  problem, before building.

### Root cause

The deploys were rejected by Vercel at config-parse time because of **`vercel.json`**
itself. Confirmed by bisecting with real production deploys:

| Commit    | `vercel.json`                                   | Result   |
|-----------|-------------------------------------------------|----------|
| `9f0fc9a` | full file minus `regions`                       | FAIL     |
| `8ce7424` | **removed entirely**                            | **PASS** |
| `47e8a2f` | restored minus `regions` and `outputDirectory`  | FAIL     |

So the failure was not the `regions` field (removing it changed nothing) but the
presence of the `vercel.json` **build-override block** as a whole — every deploy with the
file present was rejected instantly (no build logs, no `bundler`/`lambdaRuntimeStats`
metadata), while removing the file produced a normal ~75s build that succeeded. Vercel
auto-detects the Next.js framework and applies the identical build/install commands and
output handling, so the file was fully redundant. The abrupt Jun 16 onset (with no repo
change) indicates Vercel tightened enforcement on these legacy `version: 2` build
overrides around that date. The exact offending key was not isolated further to avoid
additional failed production deploys; removing the redundant file is the correct fix.

---

## Changes

1. **Fix (`vercel.json`)** — deleted the file. All of its keys (`buildCommand`,
   `installCommand`, `devCommand`, `framework`, `outputDirectory`, `regions`, `public`)
   were Vercel's auto-detected Next.js defaults, and the `version: 2` build-override block
   was what Vercel rejected. With the file gone, deploys build and ship normally.

2. **Deploy verification (`utils/update-research-cron.sh`)** — the cron previously
   equated "the headless agent exited 0" with success, which only means *committed &
   pushed*. Added `verify_deployment()`: after a successful agent run that produced a new
   commit, it polls the GitHub "Vercel" commit status (via `gh`, no Vercel CLI auth
   needed) until the deploy reaches `success` / `failure` / `error` or a 10-minute
   timeout. A failed deploy now writes a `FAILED` marker, records `deploy_failed` in
   `history.jsonl`, sets `STATUS.md` to failure, and exits non-zero. Inconclusive checks
   (gh unavailable / timeout) warn but do not fail the run.

3. **Dependency refresh (`package.json`, `package-lock.json`)** — updated all packages to
   latest within a safe range. Held back four invasive majors to avoid breaking a live
   site autonomously: **Tailwind v4** (CSS-first migration), **@notionhq/client v5**
   (incompatible with `notion-to-md@3`), **Zod v4** and **TypeScript 6**. `lucide-react`
   pinned to the latest `0.x` (`0.577.0`) because v1 removed brand icons (`Github`,
   `Youtube`) that the portfolio uses.

4. **Lint toolchain (`eslint.config.mjs`, `package.json`)** — `next lint` was removed in
   Next 16.2, so the `lint` script now calls `eslint .`. The config was migrated off the
   `FlatCompat` shim (which crashes under ESLint 9) to `eslint-config-next`'s native flat
   config. Fixed the pre-existing issues this surfaced: `Math.random()` id in `JsonLd`
   (→ `useId()`), derived-state-in-effect in `ClientBlogList` (→ `useMemo`), and a
   `next-themes` type-import path.

5. **Cadence (`research-update.config.json`)** — daily updates are unnecessary for a
   citation count, so the schedule changed from `0 2 * * *` (daily 02:00) to
   `0 9 * * 0` (Sundays 09:00 America/New_York) and the crontab was re-synced. The local
   crontab + headless `claude` design was kept deliberately: the job drives a Playwright
   browser to screenshot Google Scholar, which needs a real browser and benefits from the
   residential IP (Scholar CAPTCHAs datacenter IPs, and the prompt aborts on CAPTCHA), so
   a cloud routine would be less reliable, not more.

## Verification

- `npm run build`, `npm run lint`, and a clean `npm ci` all pass locally on Node 22
  (matching the Vercel runtime).
- `verify_deployment()` was tested against historical commits: it correctly reports
  `failure` for `b655e17` and `success` for `a798d48`.
- Production deploy confirmed READY after pushing the fix (see commit history / Vercel).
